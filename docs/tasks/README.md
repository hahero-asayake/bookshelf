# bookshelf 実装計画 (2026-06 ラウンド) — マスタープラン

残バックログを完遂するためのウォーターフォール計画。**実装 AI はまず [COMMON.md](COMMON.md) を読み、次に担当タスクの指示書だけを読んで作業する**こと。

## 確定済みの設計判断 (2026-06-12 ヒアリング)

| 論点 | 決定 |
|---|---|
| GitHub トークン期限 (8h で失効) | **refresh_token による自動更新を実装** (Worker が client_secret を注入) |
| PWA アイコン | **AI が複数案生成 → ユーザ選択**。方向性: 「Kindle アイコン風 (フラットなシルエット人物 + グラデ地)。ただし読む子供ではなく**本棚から本を取る子供**」 |
| 公開機能の配信構成 | **公開専用 repo `bookshelf-public`** に public スナップショットを push。private データは構造的に公開されない |
| Google Drive / Dropbox | **両方実装する** |
| 画像のみビュー + ツールバー | **ツールバーを 4 動詞に再構成** (検索 / 絞り込み funnel / 表示 sliders / 選択)。表示 popover に「表示形式 (表紙/画像/リスト) + 並び順 + 表紙サイズ」を集約 |
| 長文メモ frontmatter | **アプリ内では非表示** (ファイルには維持、保存時に updated を自動更新) |
| 品質基盤 | テスト/CI + toast 統一 + bookshelf.js 分割 (分割はテスト整備後の最終盤) |
| 実装運用 | **タスク完了毎に自動 push** (検証全通過まで push 禁止) |

詳細な理由は vault の `08_意思決定記録.md` (ADR-021〜025)。

### 順序の理由

1. **T01 が最優先**: 8 時間ごとの再認証は毎日の実害。かつ T07/T08/T09 が認証基盤に依存する
2. **T02〜T04 は独立した小タスク**: 早めに UX 改善を出す。T03 は UI の土台変更なので T05 のテスト対象に含めたい → T05 より前
3. **T05 (テスト/CI) は大物 (T07〜T09) の前**: アダプタ・公開はドメインロジックを触るため、リグレッション網を先に張る
4. **T12 は最後**: 移動リファクタは全機能完成 + テスト green の状態でのみ安全

## 各タスクの進め方 (実装 AI 向け)

1. [COMMON.md](COMMON.md) を読む (毎セッション必須)
2. 担当タスクの指示書を読む。「事前手作業」が未完ならユーザに依頼して**着手しない**
3. 指示書の手順に従い実装 → **受け入れ基準を全て満たすまで検証** → 設計書同期 → commit → push → この README の状態欄を `[x] 完了 (日付)` に更新して同 commit に含める
4. 指示書と実態が食い違う場合 (ファイルが無い・行番号がずれた等) は、**勝手に解釈を広げず**、grep で実態を確認して指示書の意図 (目的セクション) に沿う最小の読み替えをする。意図レベルで矛盾する場合は停止してユーザに確認
